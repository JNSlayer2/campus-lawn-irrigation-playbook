# Campus Lawn Irrigation Playbook

用這個 skill 處理以下任務：

- 校園草皮噴灌時間查詢
- 夏季 / 春季 / 秋季 / 冬季的預設灑水排程
- 某一天遇到雨天、課程、活動、工程時要不要停灌或改時段
- 草皮太濕時要如何縮短時間、減少開關次數或延後恢復
- 把「固定排程 -> 依情境調整」整理成穩定可重用的決策規則

## Read First

1. `references/irrigation-policy.yaml`
2. `references/openclaw-handoff-prompt.md` only when you need to brief another agent or initialize OpenClaw with this skill

## Core Workflow

1. 先判斷季節：`summer`、`spring_autumn`、`winter`
2. 再判斷星期與受影響區域
3. 辨識是否有情境事件：`daily`、`activity`、`engineering`、`saturated_turf`
4. 依照優先序覆蓋預設排程
5. 用固定格式回答：原定排程、調整建議、原因、恢復條件

## Decision Priority

當多個情境同時發生時，使用以下優先序：

1. 工程或安全封鎖
2. 雨天或草皮過濕
3. 活動或課程占用
4. 日常例外
5. 季節預設排程

高優先序會覆蓋低優先序。若只有部分草皮受影響，未受影響的分區可維持原排程。

## Situation Rules

### `daily`

- `中午班級曬教具`
  - 取消受影響區域的中午時段
  - 若當天仍需補水，改到下一個不衝突的常規時段
  - 班級與教具未撤離前不要重開
- `雨天`
  - 小雨：取消同時段或最近一個時段的灑水
  - 大雨或連續降雨：停灌直到草皮不積水、不黏腳
  - 恢復時先用 50% 時長試跑 1 到 2 次
- `溜冰課`
  - 停止與課程重疊的時段
  - 優先避免課前 2 小時內把場地灑濕
  - 若只影響部分區域，其他區域照常

### `activity`

- `母親節`、`校慶`、`戶外節日活動`
  - 活動搭設開始後，受影響區域停灌
  - 活動當天停灌；若前一天傍晚已進場搭設，連同前一個晚間或凌晨時段一起取消
  - 活動後先巡查壓實與積水狀況，再決定是否恢復

### `engineering`

- `割草`
  - 割草進行中停灌
  - 受影響區域建議略過割草前最近一個時段，避免濕草屑堆積
  - 清場完成後再恢復
- `壓草皮`
  - 避免長時間連續灑水
  - 優先減時，不優先加次數
  - 壓實完成後，依土壤表面乾濕度決定是否用 50% 時長恢復
- `大型工程車進場`、`鉅樹工程`
  - 視為高優先序封鎖，受影響區域全面停灌
  - 人員、車輛、材料未撤離前不要恢復
  - 若只有局部施工，其他區域可維持原排程

### `saturated_turf`

草皮過濕時，優先保護草皮，不追求照表全開：

- 先取消下一個時段，或把下一次時長降到 50%
- 同一天的開關次數降到 `0-1` 次，避免反覆補水
- 若有積水、踩踏明顯回滲、表土發亮發黏，維持停灌
- 連續 1 到 3 個時段恢復正常前，先用短時段觀察

## Output Contract

預設用這個順序回答：

1. `判斷結果`
2. `原定排程`
3. `調整建議`
4. `原因`
5. `恢復條件`

如果使用者只想要快速答案，可壓縮成：

- 結論
- 要停或要改到哪個時段
- 一句提醒

## Guardrails

- 不要捏造使用者未提供的新時段
- 若使用者沒有指明日期，至少要先判斷 `season + weekday + scenario`
- 若工程或活動只影響局部，回答時要明確標示「受影響區域」與「未受影響區域」
- `winter` 未提供凌晨時段時，預設視為冬季沒有 `02:00` 噴灌，除非之後有新規則覆蓋
